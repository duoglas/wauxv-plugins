# 红包伸手党 — 合并后判定重设计（事后宽限 + in-process 快照）

> 日期: 2026-06-09
> 状态: 设计待评审（用户已拍板"合并后再做、设计+测试用例要完备"）
> 取代: §25.10（v1.11.5 检测时快照 + 双判据）在**合并后**的实现；v1.11.5 作为合并前过渡保留
> 前提: GA+RP 合并工程（merged/）完成到伸手党判定与 recordSpeak/L3 缓冲同插件（同 BeanShell 解释器、共享全局）

## 0. 为什么要重设计（三类误判，同一病根）

真机实证（2026-06-09），v1.11.5 把**红包前/附近发过言的活跃成员**误判伸手党，三个例子三个失败模式、同一病根：

| 例子 | 失败模式 |
|---|---|
| 郑州清风 17:42 发言被警告（17:41 重试包基准） | `packetMs` = 红包检测/发出时刻；**重试包冻结在原始发出时刻**，之后发言不算 |
| 一批发过言的人被警告 | `freeloaderDecide` 的 `curLs <= packetMs` 上界把"红包后还在说话"的人排除 + 快照 flush 延迟 |
| 沟通"红包前一句正在说话"被警告 | 快照在**红包检测那一刻**拍，红包前最后 ~30s 的发言还在 GA 内存缓冲、未落表 → 快照漏掉 |

**根因（三合一）**：
1. 微信不给 per-person 领取时刻 → 只能用红包检测时刻 `packetMs` 当全员基准。
2. GA `speak` 表只存**最近一次** `last_speak`（无历史）→ 抢后发言会覆盖抢前值。
3. RP 跨插件只能读 GA 已落表数据，GA **每 30s 才异步 flush** → RP 永远慢一个 flush 周期，红包附近刚发的言看不到。

v1.11.0→1.11.5 每补一个洞冒出另一个 = 架构问题，靠跨插件读异步表救不了。

## 1. 合并后多出来的能力（这是重设计的地基）

合并后判定逻辑与 `recordSpeak` / `l3Flush` / `L3_dirtyLs` 缓冲在**同一插件、同一解释器**，于是：

- 判定/快照前可**直接调 `l3Flush()`**（GA 现成的 flush-before-read，原本只 GA 自己能用），把内存缓冲逼进表再读；或直接读 `max(表 last_speak, L3_dirtyLs 缓冲值)`。
- **flush 延迟彻底消失**（病根 #3 解决）。不再"慢 30s"。

病根 #1（无领取时刻）仍在 → 仍用 `packetMs` 基准；但配合下面的**事后宽限**，把"红包附近发言"都算活跃，#1 的副作用被吸收。

## 2. 判定语义（用户拍板，2026-06-09 细化）

> 伸手党 = 抢了拼手气红包、且在 **`[packetMs − N分钟, packetMs + 事后宽限G]`** 这个窗口内**完全没在本群发过言**的人。

- **N**（窗口，默认 30min，per-group `rp_freeloader_win_<gid>`）：红包**之前**要回看多久。
- **G**（事后宽限，新增 `rp_freeloader_grace_after_<gid>`）：红包**之后**多久内发言仍算"参与了红包附近的交流"= 活跃。用户拍 **D1=3s**（评审有异议，见 §6 D1 风险）。
- **抢后很久（>G）才发言不洗白**：依旧不算活跃（保留原"抢后不洗白"精神，只是给一个小的事后宽限把"红包一出就冒泡"的正常人放出来）。

非对称窗口（前看 30min、后宽限几秒）= "长期有没有参与" + "红包当下的即时反应"。

### 2.1 新人豁免（D2，2026-06-09 拍板）
**从未发言（窗口内无发言）的领取者 → 判伸手党，但"新人"豁免**。"新人" = 进群时间在 **新人豁免期 EXEMPT（默认 1 天，对齐 GA `#潜水` 的 `LURK_NEWMEMBER_EXEMPT_MS`）** 内。
- 判据补一维：`伸手党 ⟺ snapLs < (packetMs − N) 且 NOT(snapFs > packetMs − EXEMPT)`。即"窗口内没发言**且不是新人**"才判。
- **数据来源即局限（评审 CRITICAL，写明）**：GA `speak.first_seen` 是"**首次被观测时间**"近似，**不是真进群时间**——来源三处：`onMemberChange` 入群事件（仅 bot 在该启用群时收得到，`recordFirstSeen`）、`enableGroupWithBaseline`（开群管那刻）、onLoad baseline（v1.20.0 对已启用群补缺失成员=onLoad 时刻）。后果：**bot 进群前就在群、且从没发言的老成员**，其 first_seen 会被 onLoad baseline 补成"较晚的观测时刻"→ 可能被误当新人**误豁免**（漏判，偏保守、不误踢，可接受）。发言者的 first_seen=首次发言时刻。**接受该近似**：宁可漏判老赖，不误踢——与本功能"误踢代价最高"一致。
- first_seen 同样走 in-process 取值（它也在 `L3_dirtySeen` 缓冲）：见 §3.1 快照 SELECT 一并带回，零额外查询。

## 3. 机制：单快照，拍在 `packetMs + G` 那一刻

核心洞察：要判"`[P−N, P+G]` 内有没有发过言"，只需在 **`T_snap = packetMs + G`** 那一刻冻结每人的 `last_speak`，然后比一条阈值。

```
判定（每个拼手气领取者 w）:
    snap = FREELOADER_SNAP[packetJobKey]               // 在 T_snap=packetMs+G 拍的整群冻结快照
    if (snap == null)            → SKIP 本包整段不判    // ★降级安全网, 见 §4.1
    (snapLs, snapFs) = snap[w]                          // w 的 last_speak / first_seen (in-process 精确)
    if (snapLs 取不到/快照无此人) → SKIP 该人不判        // ★绝不把"无数据"当伸手党
    isNewMember = (snapFs != null && snapFs > packetMs − EXEMPT)
    伸手党  ⟺  snapLs < (packetMs − N分钟)  且  !isNewMember
```

为什么对（latest-only 也成立）：
- 若 w 在 `[P−N, P+G]` 任意时刻发过言 → 到 `T_snap` 为止他的"最近发言"≥ P−N → `snapLs ≥ P−N` → **不判**。✓
- 若到 `T_snap` 为止最近发言 < P−N 且非新人 → 整个窗口没发言 → **判**。✓
- 抢后 >G 的洗白发言发生在 `T_snap` **之后** → 不在快照里 → 不污染 → 仍判。✓
- "窗口内发过 + 抢后很久又发"：快照在 `T_snap` 拍，抓到的是窗口内那条（晚的还没发生）→ 活跃。✓
- **从未发言但新人**：`snapLs` 无/旧但 `snapFs` 在 EXEMPT 内 → 豁免不判。✓

### 3.1 时序（全程后台/延迟任务，零 onHandleMsg 热路径）
```
t0 = packetMs           红包检测入队（热路径只入队，不变）
t0 + G                  ← 拍快照: 先查该 job 是否已终结(放弃/无领取者)——已终结则不拍不 put;
                          否则 l3Flush() 把缓冲落表 → SELECT wxid,last_speak,first_seen FROM speak WHERE grp=?
                          → 冻结进 FREELOADER_SNAP[packetJobKey] (整群一张 HashMap, 值=(ls,fs))
t0 + 开包延迟(~90s)      判定: 逐领取者读 FREELOADER_SNAP → 按 §3 判定式（含新人豁免 + 无快照 SKIP）
```
约束与夹取（评审 MAJOR）：
- 快照必须**在判定前拍好**：`G < 开包延迟的抖动下界`。开包延迟有 ±25% 抖动（`hbJitter`），下界 = `delaySec*0.75`。**运行期夹取** `G_eff = min(G配置, 开包延迟下界 − 安全余量)`；若夹取后 `G_eff <= 0` 或快照线程未及时拍好 → 判定读不到快照 → 按 §4.1 整包 SKIP（不误判）。
- **拍照 vs 终结竞态**：红包在 `[t0, t0+G]` 之间被放弃/读不到领取者时，拍照延迟任务**起手先检查 job 终结标记**：已终结则**直接 return 不 put**（防 put 一个无人删的孤儿快照泄漏）。终结路径的 remove 仍保留（幂等，remove 不存在的 key 无害）。
- SELECT 一次带回 `last_speak,first_seen`（同一行），新人豁免**零额外查询**。

### 3.2 快照取值（二选一，实现期定）
- **A. flush-before-snapshot**：`T_snap` 时 `l3Flush()` 后 `SELECT … FROM speak`。简单、复用现成 l3Flush。
- **B. 读 max(表, 缓冲)**：`T_snap` 时持 `L3_LOCK` 短读 `max(表 last_speak, L3_dirtyLs[grp][w])`，免一次写库。
- 两者都零 flush 延迟。倾向 A（直白、与 GA flush-before-read 同范式）。

## 4. 不变项（沿用现状，别动）

- 警告交 GroupAdmin 走原 `doWarn` 流程（合并后是**同插件直调** `doWarn`，不再发群消息命令——见合并工程 P3）。
- 恒定 summary 日志（§25.10 起）：`freeloader judged/warned/active/skip_norow/excluded/capped/snap`，脱敏（计数 + wxid 尾4）。**新增**：日志区分 `snapLs` 落在 前窗/同时/后宽限/超窗 哪段，便于复盘（替代 v1.11.5 那个"curLs=old"分不清 >packetMs 的烂日志）。
- 快照生命周期配平：put（`T_snap` 拍，且 job 未终结才 put）↔ remove（判定 finally + 所有判定线程未起的终结路径兜底 + `>200` backstop）。快照值由 v1.11.5 的单值 `ls` 变为 `(ls, fs)`。

### 4.1 ★降级安全网（评审 CRITICAL — 替代 v1.11.5 双判据兜底）
新设计判定退化成单条 `snapLs < thr`，**绝不能让"没数据"被当成"小于阈值"误判伸手党**。明文规定，三处必须 SKIP（不警告、不踢）：
1. **整包快照缺失/拍照失败**（`FREELOADER_SNAP[jobKey]==null`，如 `l3Flush`/SELECT 抛异常、G 夹取后没拍成）→ **整包 SKIP**，summary 记 `snap=fail`。
2. **快照里无此领取者 / `snapLs` 取不到（null）**→ 该人 SKIP（计 `skip_norow`）。注意：`snapLs==NULL`(有行但 last_speak 为空) 与"从未发言"区分——`last_speak IS NULL` 走新人豁免判据（看 first_seen），不直接当伸手党。
3. **DB 不可用 / 任何异常**→ SKIP。
> 红线：本功能是踢人上游，**宁可漏判一个伸手党，绝不误踢一个正常人**。任何"拿不准/没数据"一律不判。

## 6. 决策（2026-06-09 用户拍板）+ 一处评审异议待复核
- **D1 — 事后宽限 G = 3s（用户拍板）**。⚠️**评审异议（MAJOR，未消解）**：本设计 §6 自身与对抗 Reviewer 均建议 ≥10s。理由：沉默 30min 的成员看到红包→点开→打字→发出"谢谢老板"真实常 >3s（手机输入法/长消息列表），G=3s 会把这类**红包后才反应的成员判伸手党并累计踢人**。缓解事实：**真正活跃的聊天者由"前 30min 窗口"兜住，G 只影响"沉默 30min、专为红包冒泡"的边界人**——对这类人 3s 是否过严是产品取舍。**记录为已知风险**：按 D1=3s 实现，但 per-group 可配 `rp_freeloader_grace_after_<gid>`，建议生产先用较大值观察、再收紧。**实现前请用户在知悉此风险后最终确认 3s。**
- **D2 — 从未发言 → 伸手党，新人（进群时间 < EXEMPT，默认 1 天）豁免（用户拍板）**。数据通道 + first_seen"首次观测"近似局限见 §2.1。`last_speak IS NULL`（有行无发言）走 first_seen 维度，不直接 SKIP。
- **D3 — 重试包基准 = 原始检测时刻（用户确认）**：`packetMs` 取 `JOB_DETECTMS`（透传），快照在"原始 P + G"拍，重试不改基准、不重拍。语义="红包刚出现那会儿你在不在场"。
- **D4 — 快照范围**：整群一张 HashMap，in-process 拍，沿用现状。无异议。

### 6.1 评审遗留需实现期落实（来自对抗审查，已并入 §3.1/§4.1/§5）
1. 新人豁免数据通道：快照 `SELECT … last_speak,first_seen`，判定式纳入 first_seen（§2.1/§3）。**否则 D2 无法实现**。
2. 降级安全网：无快照/失败/无此人 = SKIP，绝不当伸手党（§4.1）。回归测 5.2#21/#22。
3. 拍照(P+G)与终结路径竞态：拍照前查终结、`G` 运行期上界夹取（§3.1）。回归测 5.2#24/#26。
4. 命令消息不记 last_speak（GA `recordSpeak` 在命令早退之后，`main.java:2213`）：实现期确认命令路径是否补记，不补则文档记为已知残留（测 5.2#13）。
