# 红包伸手党治理 — 设计文档

> 日期: 2026-06-09
> 状态: 设计待评审 (brainstorming 产出, 评审通过后转 writing-plans)
> 涉及: RedPacketStats (生产+判定) + GroupAdmin (警告执行) 两插件, 轻量打通不合并

## 1. 目标与硬约束

**目标:** 治理"红包伸手党"——长期潜水、只在红包来时冒出来抢、平时不参与群聊的人。规则:抢红包时若**本人在抢到时刻前 N 分钟(默认 30)内未在群发言**,则给一次警告;警告累计到上限(复用群管现有 `wk_`,默认 10)自动踢出。

**硬约束(决定形态):** bot **无法物理阻止**别人抢红包(控制不了别人手机的微信)。因此"活跃才能抢"只能做成**事后检测 → 警告 → 累计踢**的威慑,不是"挡住不让抢"。

**架构决策(已确认):** 不合并两插件。RedPacketStats 负责判定 + 发警告命令;GroupAdmin 走它**原有警告流程**执行。两插件通过"一条真实群消息(警告命令)"通信,不引入共享事件库。

## 2. 数据流

```
RedPacketStats 后台反射领取明细 (NewDetailUI, 非 onHandleMsg 热路径)
  1. hbExtractTriple 增提每个领取者的「领取时刻」(需真机探测字段, 见 §6)
  2. 本群伸手党开关开 + 群管在该群已启用 时, 对每个抢包人:
       读 groupadmin.db 的 speak 表 → 该 wxid 的 last_speak (只读查)
       若 last_speak 为空 或 last_speak < 抢到时刻 − N分钟 → 判定伸手党
  3. 伸手党 → sendText(群, "[AtWx=wxid] 警告 {N}分钟内未发言")

GroupAdmin onHandleMsg (原流程, sender = bot = owner, 已授权)
  4. @ 解析[改1]: 动作词="警告", 理由="{N}分钟内未发言"
  5. doWarn[改2]: 收 reason → 计数+1(同一计数) → @通知带理由 → 到 wk_ 上限自动踢
```

例:`⚠️ 警告 2/10 — 张三 还剩 8 次 · 30分钟内未发言`

## 3. 组件职责

### 3.1 RedPacketStats (判定 + 发命令)
- **新增 A — 提领取时刻:** `hbExtractTriple` / 领取者提取除 `{昵称, 金额, wxid}` 外, 再提一个「领取时刻」字段。字段名/格式未知, 需先真机探测(§6)。提不到 → 该人本次跳过, 不误判。
- **新增 B — 读 last_speak:** 后台反射线程里, 只读打开 `…/GroupAdmin/groupadmin.db`, 查 `SELECT last_speak FROM speak WHERE grp=? AND wxid=?`。群管后台 flush 周期 30s, 故读到的 last_speak 至多 30s 陈旧, 对 30 分钟窗口可忽略。DB 打不开/查不到 last_speak → 视为"无发言记录"按伸手党, 还是跳过? **取保守: 查不到 last_speak 记录(该群无此人 speak 行)→ 跳过不判**(避免群管没数据时误伤); 有行但 last_speak 为空(从未发言)→ 判伸手党。
- **新增 C — 判定 + 发警告:** 本群伸手党开关(红包侧 per-group key)开启时, 对超窗口者发 `[AtWx=wxid] 警告 {N}分钟内未发言`。N = 该群窗口配置。
- **不碰:** onHandleMsg 热路径、检测/入队/worker 主干、统计/导出/定时。新增逻辑全在后台反射线程。

### 3.2 GroupAdmin (警告执行, 两小改)
- **改1 — @ 解析带理由:** 现 @ 路径取**最后一个 token** 当动作词。扩展(仿现成 `@X 管理员 N` 两段解析): 当最后 token 不是动作词、但**前一个 token 是 "警告"** 时, 动作=警告、理由=最后 token。`dispatchAction` 透传 reason 给 `doWarn`。
- **改2 — doWarn 带理由:** `doWarn(groupId, sender, target, reason)` 增收 reason, 把它拼进现有 @ 通知文案(`… 还剩 X 次 · {reason}`)。**不新增存储键、不动 `警告名单/警告详情`**(理由只为当场告诉本人, 非归档)。
- **改3 — 自动来源静默跳过豁免对象:** 当 reason 非空(=自动伸手党警告)且 target 命中 白名单/保护/管理/owner(`canActOn` 假 或 `isWhitelistAny` 真)时, `doWarn` **静默 return**(不发"❌ 权限不足"/"🛡 白名单"那类报错), 避免刷屏。手动警告(无 reason)行为不变, 仍给提示。
- **复用不动:** 计数 `wn_`、上限 `wk_`(默认10)、到顶自动踢 + 踢成功清零、白名单免警告。伸手党警告只是 doWarn 的一个新调用来源。

## 4. 配置

| 项 | 位置 | 默认 | 说明 |
|---|---|---|---|
| 伸手党治理开关 | RedPacketStats per-group key (如 `rp_freeloader_on_<gid>`) | 关 | 每群独立, 仅开启群判定+发警告 |
| 窗口 N 分钟 | RedPacketStats per-group key (如 `rp_freeloader_win_<gid>`) | 30 | 抢到时刻前 N 分钟无发言即伸手党 |
| 警告上限/踢 | 复用 GroupAdmin `wk_<gid>` | 10 | 不另设, 走现有警告系统 |

开关/窗口入口: RedPacketStats 文字命令 + 设置 Dialog 子页(沿用现有子页骨架风格)。

## 5. 依赖与前提
- 该群 **GroupAdmin 已启用**(才有 speak/last_speak 数据)。
- 该群 **RedPacketStats 伸手党开关已开**。
- bot 账号 = owner(`getLoginWxid`), 对 GroupAdmin 警告命令有权限(现状成立: owner 本就用 bot 账号发管理命令)。

## 6. 真机探测阶段 (领取时刻字段, 一次性)
"实际抢到时刻"是用户明确要求的判定基准。微信领取者模型对象里应含领取时间(秒/毫秒), 但字段名/类型未知。实现前先做一次真机探测:
- 在 `hbExtractTriple` 拿到的领取者元素对象上, 反射枚举字段/getter, 脱敏打印候选时间值(long 且接近当前 epoch 的字段), 比对真实领取时间定位字段。
- 探测脱敏: 只打印字段名 + 是否 long + 值是否落在合理 epoch 区间, **绝不打印 wxid/金额/昵称**。
- 定位后写死字段访问 + 兜底(取不到 → 该人跳过)。版本脆弱: 微信升级后若字段漂移, 退化为"取不到→跳过", 不崩、不误判。

## 7. 限制 (已知, 向用户明示)
- **只管拼手气红包:** 均分(等额)包现被判普通直接跳过, 不反射明细 → 抢均分包的人管不到。
- **只管被领完的包:** 未领完的包走重试/放弃, 不进领取明细。
- **领取时刻字段版本脆弱:** 探测得到才有, 提不到 → 该人本次跳过不误判。
- **last_speak 至多 30s 陈旧**(群管 flush 周期), 对 30 分钟窗口无影响。

## 8. 风险与 QA
- **破坏性动作(自动踢):** 高风险。VERIFY 阶段走 Santa 双独立 Reviewer。误踢防线: 复用 `canActOn`/白名单/保护/owner 豁免 + 自动来源静默跳过 + last_speak 查不到则跳过。
- **热路径(C-PERF-01):** 全部新增在后台反射线程, onHandleMsg 一行不加。
- **反检测:** bot 发警告命令/@消息 = 现有群管常规行为(已验证不触发风控), 不新增暴露面。
- **跨插件读 DB:** RedPacketStats 只读打开 groupadmin.db, 用独立连接, SQLite 多连接安全; 打不开则跳过判定(降级不报错)。

## 9. 构建顺序 (供 writing-plans 细化)
1. GroupAdmin 改1+改2+改3(@解析带理由 / doWarn 收 reason / 自动来源静默跳过) — 可独立部署验证(手动发 `@X 警告 测试理由` 看通知带理由 + 豁免对象静默)。
2. RedPacketStats 真机探测领取时刻字段(§6)。
3. RedPacketStats 新增 A/B/C(提领取时刻 / 读 last_speak / 判定发警告) + per-group 开关与窗口配置。
4. 端到端真机验证 + Santa 双 Reviewer。
